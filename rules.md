# 지침

## 작업 진행 원칙

### 핵심원칙(대원칙)

- 별도의 지시가 없는 한, 수정 가이드를 제안하여 명시적 승인을 받는 방식으로 진행

### 개발 프로세스 관리

- 개발 플랜을 먼저 작성한다
- 작성한 플랜 내의 프로세스에 따라 개발을 진행하되 플랜이 여러 프로세스로 구성될 경우, 각 프로세스가 끝날 때마다 진행한 내용을 요약하고 다음 프로세스의 진행을 요청한다. 이때 위에 기재한 핵심원칙(대원칙)을 따른다.
- 진행요구에 승락('진행해' 등)이 있을 경우 진행한다.

### 명시적 승인 필수

- "해줘", "진행해", "실행해", "만들어줘" 등 **명확한 지시**를 할 때만 작업 진행
- "할 수 있나?", "가능한가?", "방법이 있나?" 같은 **질문**에는 가능 여부만 답변하고 대기
- 예시:
  - ❌ "문서를 한국어로 생산할 수 있나?" → 바로 변환 작업 진행
  - ✅ "문서를 한국어로 생산할 수 있나?" → "네, 가능합니다" 답변 후 대기

### 확인 후 실행

- 파일 수정, 삭제, 생성 등 중요한 작업은 반드시 확인 필요
- Git 백업 상태를 항상 확인
- 중요한 변경 전에는 알림

## 답변 원칙

### 간결성

- 항상 간결하고 핵심 중심으로 작성
- 불필요한 서론, 인사말, 반복 설명 생략
- 바로 핵심부터 답변
- 약어는 뒤에 전체 용어를 괄호로 표기. (예시 : DB(database))
- 절대 코드를 작성하여 설명하지 않는다.

### 답변 길이 기준

- **사실 질문**: 1-2문장
- **개념 설명**: 3-5문장
- **복잡한 주제**: 짧은 단락 1-2개 (5-8문장)

### 예외

- "자세히", "상세히", "깊이 있게" 요청 시: 필요한 만큼 충분히 설명

## 문서 작성

### Artifact 문서

- 모든 `.md` 문서는 **한국어**로 작성
- 간결하고 명확하게 작성
- 불필요한 설명 최소화

### 주석

- 코드 주석은 **한국어**로 작성
- 복잡한 로직에만 추가
- 자명한 코드에는 주석 불필요

## Flutter/Dart 개발 규칙

### 아키텍처

- Clean Architecture 준수 (UI, Domain, Data 레이어 분리)
- 단일 책임 원칙: 각 클래스/함수는 하나의 명확한 목적만
- 의존성 주입 활용

### 코드 스타일

- **네이밍**:
  - 클래스: `PascalCase` (예: `UserService`)
  - 함수/변수: `camelCase` (예: `getUserData`)
  - 상수: `UPPER_SNAKE_CASE` (예: `MAX_RETRY_COUNT`)
  - Private: 언더스코어 시작 (예: `_privateMethod`)
- **함수 길이**: 최대 50줄 이하, 넘으면 분리
- **파일 구조**: 관련 기능끼리 그룹화

### 에러 처리

- 모든 비동기 함수에 `try-catch` 필수
- 사용자 친화적 에러 메시지 제공
- 로깅: 개발/프로덕션 환경 구분

### UI/UX

- 반응형 디자인: 다양한 화면 크기 고려
- 로딩 상태 명확히 표시
- 에러 상태 처리
- 일관된 디자인 시스템 (색상, 폰트, 간격)

### 보안

- API 키는 환경변수 또는 Cloud Functions에 저장
- 민감 데이터는 암호화
- 권한 체크 철저히

### 성능

- 불필요한 rebuild 방지 (`const` 활용)
- 이미지 최적화
- Lazy loading 활용

## Git 관리

- 중요한 작업 전 백업 브랜치 생성
- 의미 있는 커밋 메시지 작성
- 작업 전 현재 브랜치 상태 확인
- 작업 마무리 전 Git에 관한 변경사항이 있는지 확인하고 있으면 Git 작업 제안

### 작업 로그 관리

- Phase 완료 시 반드시 다음 순서로 진행:
  1. Git 커밋 (의미 있는 커밋 메시지 작성)
  2. 작업 로그 문서 생성 (`.agent/work_log_YYYYMMDD_작업제목.md`)
  3. 필요시 작업 로그 문서도 커밋

- 작업 로그 문서 구조:
  - 기본 정보: 날짜, 커밋 해시, 브랜치, 작업자
  - 작업 개요: 한 문장으로 핵심 요약
  - 변경 파일: 수정된 파일 목록
  - 상세 작업 내용: Step별 상세 설명
  - 개선 효과: 작업의 결과와 영향
  - 통계: 변경 파일 수, 추가/삭제 줄 수
